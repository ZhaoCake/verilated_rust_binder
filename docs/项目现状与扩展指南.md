# 项目现状与扩展指南

本文档说明两件事：

1. 这个仓库当前已经完成了什么。
2. 如何在现有流程中新增 SystemVerilog 文件与 Rust 侧仿真代码。

## 1. 当前项目已经完成了什么

这个仓库已经打通了完整自动链路：

`SystemVerilog RTL -> Verilator C++ 模型 -> 自动生成 C ABI + Rust API -> Rust 示例驱动仿真`

当前已具备的能力：

- 支持用 Rust 直接驱动仿真，不再依赖 C++ testbench。
- `build.rs` 会自动扫描 `rtl/` 下所有 `.sv/.v` 文件参与 Verilator 编译。
- 通过环境变量 `VERILATOR_TOP` 选择顶层模块（默认 `top`）。
- 自动调用 `scripts/gen_verilator_binder.py`：
  - 解析 `V<top>.h` 端口定义
  - 生成 C++ bridge（`vrb_new/eval/final/get/set`）
  - 生成 Rust `SimModel` 封装（`set_xxx()/xxx()`）
- 自动执行 Verilator 生成的 Makefile，编译 `V<top>__ALL.a` 并链接到 Rust。
- 已有两套可运行示例：
  - `top.sv` + `rust/examples/counter.rs`
  - `adder_top.sv` + `rust/examples/adder.rs`
- 已新增一套时序示例：
  - `pulse_counter_top.sv` + `rust/examples/pulse_counter.rs`
- 已新增一套嵌套模块示例：
  - `toggle_cell.sv` 被 `toggle_pair_top.sv` 实例化
  - `toggle_pair_top.sv` + `rust/examples/toggle_pair.rs`
- 已新增一套 DPI import 示例：
  - `dpi_adder_top.sv` 通过 `import "DPI-C"` 调用 Rust 函数
  - Rust 侧导出实现在 `rust/src/dpi.rs`
- 已支持 Rust 原生测试：
  - 可在 `rust/tests/*.rs` 中直接使用 `#[test]`
  - 可通过一次生成多个 top 的绑定来支撑集成测试
- 顶层命令封装在仓库根 `Makefile`：
  - `make sim` / `make sim-counter`
  - `make sim-adder`
  - `make sim-pulse-counter`
  - `make sim-toggle-pair`
  - `make sim-dpi-adder`
  - `make rust-sim TOP_MODULE=... RUST_EXAMPLE=...`
  - `make rust-test TOP_MODULES=...`

## 2. 当前约束与注意事项

自动绑定生成器目前仅支持：

- 标量端口（input/output/inout）
- 位宽 `<= 64`

不支持：

- `>64-bit` 端口（例如 Verilator 头文件里的 `VL_*W`）

如果使用超宽端口，生成阶段会直接报错并停止。

## 3. 如何新增一个 SystemVerilog 模块并接入仿真

下面是推荐流程。

### 步骤 1：新增 RTL 文件

把新模块文件放到 `rtl/` 目录，例如：

- `rtl/my_accum_top.sv`

说明：`build.rs` 会自动收集 `rtl/*.sv` 与 `rtl/*.v`，通常不需要手工改源文件列表。

### 步骤 2：确认顶层模块名

确保你的顶层模块名明确，例如：

- 模块名：`my_accum_top`

后续运行时要把它传给 `TOP_MODULE`（内部映射到 `VERILATOR_TOP`）。

### 步骤 3：写 Rust 驱动示例

在 `rust/examples/` 下新增示例文件，例如：

- `rust/examples/my_accum.rs`

可参考最小模板：

```rust
use verilated_rust_binder::my_accum_top::SimModel;

fn main() {
    let mut dut = SimModel::new();

    // 以下方法名会由端口自动生成。
    // 例如端口 clk/rst_n/en/out 可能对应:
    // dut.set_clk(...), dut.set_rst_n(...), dut.set_en(...), dut.out()

    dut.eval();
    println!("custom simulation done");
}
```

### 步骤 4：运行

在仓库根目录执行：

```bash
make rust-sim TOP_MODULE=my_accum_top RUST_EXAMPLE=my_accum
```

如果只是先检查编译链路：

```bash
make rust-check TOP_MODULE=my_accum_top
```

### 步骤 5（可选）：给 Makefile 增加快捷目标

如果你希望像 `make sim-adder` 一样有快捷命令，可以在根 `Makefile` 增加类似规则：

```makefile
sim-my-accum: TOP_MODULE = my_accum_top
sim-my-accum: RUST_EXAMPLE = my_accum
sim-my-accum: rust-sim
```

这样就可以直接执行：

```bash
make sim-my-accum
```

## 4. 如何新增 Rust 项目（两种方式）

“新增 Rust 项目”常见有两种诉求：

- A. 在当前 crate 下新增一个仿真入口（推荐）
- B. 新建一个独立 crate（进阶）

### A. 在当前 crate 下新增仿真入口（推荐）

这是最轻量方式，本质上是新增 `rust/examples/*.rs`：

1. 新建示例文件，例如 `rust/examples/foo.rs`
2. 使用对应 top 的显式路径编写驱动逻辑，例如 `use verilated_rust_binder::<top_name>::SimModel;`
3. 执行：

```bash
make rust-sim TOP_MODULE=<你的顶层模块> RUST_EXAMPLE=foo
```

优点：

- 不需要改 `Cargo.toml`（Cargo 会自动识别 examples）
- 复用现有 `build.rs` 自动生成绑定流程

## 4.1 如何使用 Rust 原生 test 框架测试 DUT

这也是当前仓库现在已经支持的能力。

### 推荐目录

把测试放在：

- `rust/tests/*.rs`

例如：

- `rust/tests/pulse_counter_test.rs`

### 使用方式

如果测试里需要多个 top，可一次性把它们交给构建脚本生成绑定：

```bash
VERILATOR_TOPS=top,adder_top,pulse_counter_top,toggle_pair_top,dpi_adder_top cargo test --manifest-path rust/Cargo.toml
```

或用 Makefile：

```bash
make rust-test
```

只运行某个测试文件：

```bash
make rust-test TOP_MODULES=dpi_adder_top RUST_TEST=dpi_adder_test
```

只运行某个测试函数：

```bash
make rust-test TOP_MODULES=toggle_pair_top RUST_TEST=toggle_pair_test RUST_TEST_FILTER=nested_modules_toggle_independently
```

### 测试代码写法

测试中直接 import 对应 top 的模块：

```rust
use verilated_rust_binder::pulse_counter_top::SimModel;

#[test]
fn example_test() {
  let mut dut = SimModel::new();
  dut.eval();
  assert_eq!(dut.count(), 0);
}
```

这样可以直接享受 Rust 原生测试框架的好处：

- `#[test]`
- `assert_eq!`
- `cargo test`
- 易于 CI 集成

### B. 新建独立 Rust crate（进阶）

如果你确实要拆成独立工程（例如多个团队/多个可执行程序隔离），可按下面做：

1. 在仓库内新建目录（例如 `rust_foo/`）并初始化 crate。
2. 复制并按需修改当前 `rust/build.rs`，保留以下关键逻辑：
   - 扫描 `rtl/`
   - 调用 `verilator`
   - 调用 `scripts/gen_verilator_binder.py`
   - 编译并链接 `V<top>__ALL.a`
3. 在新 crate 的 `src/lib.rs` 中保留：

```rust
include!(concat!(env!("OUT_DIR"), "/binder.rs"));
```

4. 在新 crate 的 `Cargo.toml` 增加：

```toml
[build-dependencies]
cc = "1.1"
```

5. 通过环境变量选择顶层运行：

```bash
VERILATOR_TOP=<你的顶层模块> cargo run --manifest-path rust_foo/Cargo.toml --example <示例名>
```

建议：除非有明确隔离需求，优先使用方式 A（新增 examples）。

## 5. 常见问题排查

- 报错：`No rtl source found`
  - 检查是否把 `.sv/.v` 放到了 `rtl/` 目录。
- 报错：找不到某个 `set_xxx()` 或 `xxx()` 方法
  - 先确认顶层模块是否正确（`TOP_MODULE` 是否与模块名一致）。
  - 再确认端口是否在该顶层模块里真实存在。
- 报错：端口宽度不支持
  - 检查是否有 `>64-bit` 端口，当前生成器不支持。

## 6. 一条命令复用模板

新增任意模块后，最常用命令模板是：

```bash
make rust-sim TOP_MODULE=<sv顶层模块名> RUST_EXAMPLE=<rust示例文件名去后缀>
```

例如：

```bash
make rust-sim TOP_MODULE=adder_top RUST_EXAMPLE=adder
```

## 6. 嵌套模块是否支持

支持。

当前仓库已经加入一个专门用于验证“模块之间有实例化关系”的例子：

- 子模块：`rtl/toggle_cell.sv`
- 顶层模块：`rtl/toggle_pair_top.sv`

其中 `toggle_pair_top` 内部实例化了两个 `toggle_cell`：

- `u_toggle_a`
- `u_toggle_b`

Rust 侧通过顶层端口进行观察和驱动，不需要关心内部实例名，使用方式仍然是：

```rust
use verilated_rust_binder::toggle_pair_top::SimModel;
```

对应测试文件：

- `rust/tests/toggle_pair_test.rs`

运行方式：

```bash
make rust-test TOP_MODULES=top,adder_top,pulse_counter_top,toggle_pair_top,dpi_adder_top
```

## 7. 如何使用 `import "DPI-C"`

当前仓库已经支持 `import "DPI-C"`，这是最常用的一类 DPI 用法。

### SystemVerilog 侧

例如：

```systemverilog
import "DPI-C" function int rust_add(input int a, input int b);
```

然后在模块逻辑中直接调用：

```systemverilog
sum = rust_add(int'(a), int'(b));
```

当前示例顶层：

- `rtl/dpi_adder_top.sv`

### Rust 侧

需要导出同名 C ABI 符号：

```rust
#[unsafe(no_mangle)]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
  a.wrapping_add(b)
}
```

当前示例位置：

- `rust/src/dpi.rs`

### 测试方式

```bash
VERILATOR_TOPS=top,adder_top,pulse_counter_top,toggle_pair_top,dpi_adder_top cargo test --manifest-path rust/Cargo.toml --test dpi_adder_test
```

或：

```bash
make rust-test TOP_MODULES=dpi_adder_top RUST_TEST=dpi_adder_test
```

### 当前边界

当前实现支持：

- `import "DPI-C" function ...`

当前还没有专门做高级封装：

- `export "DPI-C"`
- DPI scope/context API
- 自动解析 DPI 原型并生成 Rust 声明
